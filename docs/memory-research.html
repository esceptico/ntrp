00The one-screen version

The field has quietly converged, and NTRP is already on the right side of the biggest decision — but it has built the wrong half well and left the right half as dead code.

What the field decided

  • Files won, graphs lost. Mem0 deleted its Neo4j mode; Cursor deprecated its opaque "Memories" for editable .md rules; Letta abandoned the MemGPT vector-DB design for MemFS (git-tracked markdown). Your instinct ("graphs are useless and hard") matches the market.
  • Storage is not the product — maintenance is. Everyone agrees on "markdown + a throwaway index." The differentiation is the machinery: write-gating, off-path consolidation, supersede-don't-delete, principled forgetting, and reflection.
  • Append & periodically compress (Karpathy's LLM-Wiki) is the canonical model — and dex/gbrain are its CRM-flavored implementation, not the idea itself.
  • Never let an LLM rewrite a whole memory blob. That causes context collapse. Edit small addressable deltas; compress with a regression guard.

Where NTRP actually stands

  • Ahead on "generic, not CRM." One flat directive|fact|source pool with open-vocabulary labels — subjects (Bicycles, MATS, Wife, Health) emerge bottom-up. Strictly better than dex/gbrain, which must swap a schema-pack to escape person/company.
  • Behind on what you care about most. The maintenance "brain" is spec'd in dead constants: REFLECTION, RETENTION, HEALTH, PROFILE_REFRESH are defined and zero are wired. There is no cross-domain dreaming, no forgetting, no health surface.
  • Not file-canonical despite your Eve preference: markdown is a read-only projection over SQLite plus a third vector DB — a three-source-of-truth design with real drift machinery.
The recommendation in one line Wire the dead maintenance brain first (no migration, ships value, deletes ~800 lines), then flip canonicality to plain markdown files (Eve/MemFS-style), using a two-zone "compiled prose + addressable timeline" page format throughout. The live plan is one build (§13): files-canonical from the start, the brain wired in the same pass, the old record-store engine retired — no phased rollout. (The C→A phasing in §09 is kept as design rationale, not the order to follow.) The cross-domain dream is the differentiator no commercial system ships, and it's the capability you literally asked for.

01Where the field converged (and why it left graphs behind)

Three independent 2025–2026 data points show the same retreat from databases toward plain files, and the same realization that the value is in the grooming loop, not the store.

The twelve points of consensus

  • File-based markdown + derived index is the settled storage answer; graph DBs are being actively removed.
  • Append-then-periodically-compress (Karpathy LLM-Wiki) is the canonical declarative model.
  • Maintenance machinery — not storage — is the product.
  • Write-gating with a HIGH bar is universal: "most exchanges contain nothing worth remembering."
  • Consolidation runs OFF the interactive path via a background / sleep-time agent with a fresh context window.
  • Conflicts → supersession, not deletion (Zep bi-temporal, gbrain flag-don't-resolve).
  • Forgetting is principled & automatic: TTL+LRU, retrieval-decay, or utility deletion.
  • Triggers converge on three shapes: every-N-turns, importance-budget threshold, or nightly cron.
  • Provenance is first-class and inline: [Source:] / (record:id) / (because of 1,5,3).
  • Two read modes emerging: cheap ranked search vs synthesized cited think that also reports gaps.
  • Edit memory as small deterministic deltas over addressable items — never whole-blob LLM rewrites.
  • Self-editing via file/string edits + git commits is replacing bespoke memory APIs.

03System deep-dives

Each one expandable. Read the first two (dex + Eve) — they're the systems you named — then dip into the rest as needed.

dex & the Karpathy LLM-Wiki lineage your reference #1

The idea (Karpathy). Instead of RAG over raw documents at query time, an LLM incrementally compiles a persistent wiki of markdown pages: append raw records as they arrive, then periodically compress them into synthesized prose. Three layers: raw/ (immutable sources) → the wiki (LLM-owned markdown: entity pages, concept pages, an index.md catalog, an append-only log.md) → a schema doc (AGENTS.md/CLAUDE.md telling the LLM the conventions).

"dex" is actually three different things, which is why it feels CRM-ish:

  • getdex.com — a literal personal CRM with hard-coded person fields. This is the rigidity you dislike.
  • davekilleen/Dex — a PARA-folder markdown "AI Chief of Staff."
  • gbrain (Garry Tan, open-source) — the deepest, most generic implementation: 146K+ pages, a swappable type taxonomy, an overnight dream cycle, and the (record:id)-style provenance NTRP already borrowed.

The page format (gbrain) — adopt this. Each entity = one living page with a stable slug-id. Two zones: compiled truth above the line (synthesized prose) and an append-only dated timeline below (- YYYY-MM-DD | Source — summary). Every fact carries inline provenance ([Source: User, …] / [Source: url] / [Source: compiled from …]) with a precedence ladder: user > compiled > timeline > external API. Beliefs are split from facts into a fenced "takes" table where each row names a holder and a weight.

What makes it CRM-ish vs generic. The smell is presuming a fixed entity taxonomy centered on people & companies, with relationship-management sections ("What They Believe", "Open Threads"). gbrain's escape hatch is schema detect/suggest — cluster the actual filesystem and let types emerge. NTRP already does this one better: no taxonomy at all, just open-vocabulary labels.

The dream cycle (the part you care about) — "the most important cron job, NOT optional." Nightly at 02:00, 8 phases: lint → backlinks → sync → synthesize → extract → patterns → embed → orphans. A cheap Haiku "is this worth processing?" verdict filters routine sessions before a Sonnet subagent writes reflections; the PATTERNS pass writes a pattern page only when ≥3 reflections support it. Safety throttles directly transferable to NTRP: cooldown_hours, content-hash idempotency, an exclude_patterns pre-LLM filter (medical/therapy), a write allow-list of slug prefixes, and a per-run --max-usd cost cap.

Forgetting uses two orthogonal axes: salience (mattering, time-free) and recency (per-prefix decay; concepts evergreen, daily/chat decay fast) — Ebbinghaus-style, reset on access.

Vercel Eve — filesystem-first agents your reference #2

The honest headline: Eve ships no memory subsystem and no consolidation. "An agent is a directory" — agent/instructions.md, agent/tools/*.ts (filename = tool name), agent/skills/* (markdown + frontmatter, loaded on demand), agent/schedules/*.ts (compile to Vercel Cron). Persistence comes from two layers: the Workflow event log (makes a session crash/deploy-resumable) and the Sandbox filesystem the model reads/writes via read_file/write_file/bash (durable via snapshot-on-stop, now GA).

Important tensionYou like Eve for the right aesthetic (filesystem simplicity) but it omits exactly the maintenance layer you care about. Take Eve's storage model and its "a schedule wakes the agent with a prompt" heartbeat primitive — do NOT model the maintenance brain on Eve, because it has none.

The transferable primitive is precise: a schedule's run() handler calls receive(channel, {message}) — i.e. it fires a turn by feeding the agent a natural-language instruction ("Summarize last week and post to the channel"). That is exactly the hook you'd use for a nightly consolidation/dream pass.

Letta / MemGPT → MemFS — self-editing memory & sleep-time compute

MemGPT (2023): treat the context window like an OS's RAM. Tiered virtual memory — main context (system instructions + a writeable "core memory" block + a FIFO queue with a recursive summary at index 0) and external context (recall storage = full message history; archival = vector store). The LLM pages info in/out via tool calls; the system raises "memory pressure" warnings at ~70% of the window and flushes at ~100%, regenerating a recursive summary.

Sleep-time compute: a background agent that shares the memory blocks and owns the editing tools — the primary agent is not given block-edit tools (one clean write gate). Fires every N=5 primary-agent steps (docs say keep N high — too frequent is expensive); processes only the new-messages delta; can use a stronger model since it isn't latency-bound.

MemFS / Context Repositories (newest, closest to your goal): git-tracked markdown memory files edited with bash tools, a system/ always-resident folder, and reflection/defrag subagents. This is the direction NTRP should converge toward.

Mem0 — the inline two-phase reconciler (NTRP's curator is this)

Unit of memory = a single NL fact string in a vector point (UUID + embedding + text + scope metadata). Two-phase LLM pipeline: an extraction phase pulls salient facts from recent turns + a rolling summary; an update phase, for each fact, retrieves the top-10 similar existing memories and asks the LLM to choose ADD / UPDATE / DELETE / NOOP — operating only on real ids shown to it (anti-hallucination). Consolidation is inline on every write, not a nightly batch. The 2026 rewrite dropped Neo4j graph mode for lightweight in-store entity linking — direct support for your "graphs are too hard" stance. NTRP's Curator is a near-identical clone, plus an offline consolidate pass and a prose projection Mem0 lacks.

Zep / Graphiti — the graph approach (and why to skip it)

A bi-temporal knowledge graph: episodic subgraph (raw verbatim) → semantic entity subgraph (deduped entities + typed edges) → community subgraph (clustered summaries). Every edge tracks 4 timestamps (when true in world / when system learned / when invalidated) so contradictions are expired, not deleted. Per-episode online pipeline: entity extraction → resolution/dedup → fact/edge extraction → temporal invalidation — each step an LLM call.

What to steal without the graphBi-temporal validity ("what did I believe on date X") and edge invalidation as the conflict model. NTRP gets ~80% of this cheaply via superseded_by + git history. The cost that makes it "hard to use": multiple LLM calls per write, a graph DB to operate, and extraction quality that gates everything.
Commercial note-stores — ChatGPT, Claude memory tool, Cursor, Copilot, Windsurf

Two philosophies: passive profile (ChatGPT saved-memories + reference-chat-history; Claude consumer) that auto-injects, vs discrete agent-written notes (Claude memory tool, Cursor, Windsurf, Copilot) retrieved/validated on demand. Most transferable:

  • Claude memory tool — the purest file model: a /memories dir with view / create / str_replace / insert / delete / rename, and a system-prompt protocol "ALWAYS VIEW YOUR MEMORY DIRECTORY FIRST… ASSUME INTERRUPTION." No scheduled job — you build the expiry. No imposed schema.
  • Copilot memory — citation-validation + a 28-day unused-fact TTL that resets on use (clean LRU forgetting), on by default 2026.
  • Cursor — deprecated opaque "Memories" for editable .cursor/rules/*.md. The market voting for files over a fact-store.
  • Security lesson: ChatGPT's memory-injection attacks (Embrace The Red) — untrusted content writing into memory is a live poisoning vector. Matters for NTRP's passive ingest.
Academic canon — Generative Agents, A-MEM, Reflexion, Voyager
  • Generative Agents (Stanford): the reference design. Append-only memory stream; retrieval = recency × importance × relevance; periodic reflection fires when summed importance > 150 (~2–3×/day), asks "the 3 most salient high-level questions," retrieves evidence, writes 5 cited insights — building a reflection tree. No schema, just timestamped NL + scores. The closest match to your goal.
  • A-MEM: Zettelkasten atomic notes; each new note links itself to related notes and triggers retroactive updates to older notes ("memory evolution"). Emergent structure, no schema.
  • Reflexion: verbal self-improvement — after a failure, write a NL lesson into a bounded buffer prepended to the next attempt. The seed of a "playbook."
  • Voyager: memory = a growing skill library of verified executable procedures, indexed by description embedding. New skills append without modifying old ones (no catastrophic forgetting).

04Continual learning in token space

How an agent gets better over time with zero weight updates — purely by curating a textual artifact. This is the mechanism behind "evolves from interactions without my direct action."

ACE 2510.04618

An evolving playbook of itemized bullets, each with an id + helpful/harmful counters. A Generator → Reflector → Curator loop: Reflector distills lessons, Curator emits a small delta, code merges it. "Grow-and-refine": append new ids, bump counters in place, lazily dedup by embedding only when over budget.

Dynamic Cheatsheet 2504.07952

A single evolving text block of distilled strategies/snippets. The useful variant — DC-Retrieval-Synthesis — stores past (input,output) pairs, retrieves top-3 by cosine, and synthesizes a fresh, task-specific cheatsheet per query rather than dumping the whole store. (This is the fix for NTRP's resident bullet-dump.)

GEPA 2507.19457

Memory = the system prompts themselves, plus a versioned pool of candidates with ancestry + a score matrix. Mutate, eval on a minibatch, and only keep it if it doesn't regress. The "regression guard" idea NTRP needs on its synthesis pass.

The one law: context collapse The dominant failure mode is brevity bias. When you ask an LLM to rewrite the whole memory each time, it silently compresses away hard-won specifics. ACE measured a single rewrite dropping 18,282 → 122 tokens and accuracy 66.7% → 57.1% in one step.

The fix (and NTRP's missing guard): treat memory as append-mostly, item-structured, edited with small deltas merged by deterministic code — and add a regression check that rejects any re-synthesis that drops size/citation-count past a threshold. Caveat from our own critic: this 18k→122 figure is one paper's anecdote, repeated as if it were a law. It's a real risk to guard against, not a proven universal — NTRP's existing synthesis already has citation-rejection; whether it empirically collapses is untested.

Empirical lesson on forgetting (from the experience-following study, 2505.16067): "add-all" memory degrades accuracy — one run dropped to 13% (2,400 records) vs 39% selective (248 records); utility/retrieval-history deletion was worth ~+10%. Caveat: domain-specific (medical reasoning), so treat the exact numbers as directional, not gospel — but the direction is unambiguous: forgetting is not optional.

Prompt shapes worth lifting near-verbatim

Poignancy score (Generative Agents)

On the scale of 1 to 10, where 1 is purely mundane (brushing teeth) and 10 is extremely poignant (a breakup, a college acceptance), rate the likely poignancy of: "{memory}" Rating: <int>

Stamped at write time; reused for both the reflection trigger and retrieval ranking.

Reflection / dream (Generative Agents)

1) Over the ~100 most recent records: "What are the 3 most salient high-level questions about the subjects above?" 2) Use each as a retrieval query → evidence 3) "What 5 high-level insights can you infer? format: insight (because of 1, 5, 3)"

Insights are written back as new records that cite their evidence → a reflection tree that compounds.

Consolidation (Anthropic consolidate-memory skill — 3 phases, adopt ~verbatim)

  1. Take stock — read what exists, what's new.
  2. Reconcile — separate durable vs dated; merge overlaps; absolutize relative dates; drop anything easily re-findable from calendar/docs; keep what's hard to re-derive.
  3. Tidy — keep the index under a size cap (e.g. 200 lines / 25KB).

Promotion gate (OpenClaw, generalized to be non-CRM): a short-term note graduates to a durable topic page only when it passes multiple gates — seen ≥3×, across ≥3 distinct contexts, composite salience ≥ threshold. This is how page types emerge from your own labels instead of a person/company schema.

Bounding file size & context — five mechanisms

A recurring worry with file-canonical memory: won't pages grow unbounded, and won't the model have to swallow giant files? No — the design never lets a file grow without bound, and never loads them all. Five mechanisms, working together:

  1. Two-zone read discipline. The long part of a page is the append-only timeline. By default only the short compiled-prose zone is read; the timeline is searched (pull the 3 relevant lines), not loaded wholesale.
  2. Per-page size budget + DEFRAG. Each page carries a limit in frontmatter; when exceeded, the maintenance pass splits it into sub-topics or merges near-duplicate pages. Target ~15–25 focused files, not 3 giant ones.
  3. Progressive disclosure. The agent's resident context holds only the filetree + a one-line description per page; it opens a full page only when relevant (the Claude-memory-tool "view the directory first" protocol). The map stays small even when the corpus doesn't.
  1. Forgetting trims the timeline. TTL-by-class + LRU tombstone old transient lines (salience floor protects important/pinned), so a page's timeline doesn't accrete forever.
  2. Hierarchical summarization (RAPTOR-style), if needed. daily → topic → domain-summary so retrieval enters at the right altitude and you never read the leaves to answer a high-level question.
Mental modelFiles = truth (small, bounded, human-readable, git'd). .index/ = a disposable cache (embeddings + FTS, rebuilt on demand from the files — embeddings never live in the markdown). The context window only ever sees compiled zones + pulled lines — never whole files. The "one giant file the model must swallow" failure mode is exactly what topic-splitting + two-zone + progressive disclosure prevent.

08Three architectural directions

A panel of three independent designs, from boldest-rewrite to smallest-diff. The recommendation (next section) sequences them rather than picking one.

Direction A

Brain-as-Folder

Memory is a git-tracked markdown dir the agent edits with the file tools NTRP already ships. Delete the SQLite store and the record-source vectors (the shared search.db keeps its transcript vectors); rebuild a thin grep+optional-embedding index on demand. Eve/MemFS, taken literally.

Novelty: inversion with deletion — flip canonicality to the files NTRP currently treats as a read-only projection, and delete the canonical DB rather than adding a layer.

Direction B

Living Playbook

Boldest. The atom is an addressable item (a line) with helpful/harmful counters + importance + links. Pages cluster & self-organize (A-MEM); an always-resident playbook of strategies/failure-modes improves in token space (ACE); cross-domain dreams author insights.

Novelty: items-as-lines-with-counters in human-readable markdown — unifies A-MEM self-linking + ACE delta-counters + gbrain two-zone pages on one git store. Also fuses fact memory with the agent's own procedural playbook.

Direction C

Pragmatic Hybrid

Smallest diff. Keep the SQLite engine, hybrid retrieval, provenance. Delete the dead claims-pipeline residue (~800 lines). Wire the four dead builtins — reflection/dream, retention, health, profile-refresh — each bound to a proven recipe.

Novelty: refusing the fashionable rewrite. The storage is already best-in-class for your goals; the gap is 100% the maintenance brain, and it's already spec'd in dead constants — resurrect it.

The page format all three share — two zones + addressable timeline

Worked example — one interaction through the loop

Tim says: "I finally swapped my daily commuter to the gravel bike — the road bike's bottom bracket is toast."

  • Hot path (write-tag): append - 2026-06-21 ^9c2 [imp:6] (src:chat 4f2a) gravel bike is now daily; road BB failed. to daily/. Importance budget += 6. Chat pays one tiny scoring call.
  • Night / budget>150 — consolidate in a worktree: FILE ^9c2 → topics/bicycles.md; SYNTHESIZE its compiled zone (with the regression guard); RECONCILE — strike the old "road bike is daily" line, rewrite it past-tense.
  • Dream: sees ≥3 recent gear-failure lines across bicycles.md + spending.md, writes to insights/: "Tim replaced/repaired three pieces of gear in June (^9c2, ^b4d, ^e11) — a possible budget signal." No one asked for that connection.
  • git commit — free, diffable, revertable changelog.
  • Next time "which bike should I take?" → recall("daily bike") hits bicycles.md, returns the compiled zone with a (^9c2) citation. No SQLite, no vector DB.

09The recommendation — sequence, don't choose

Ship C first (the maintenance brain, no migration, deletes ~800 lines), then converge on A (filesystem-canonical, what you asked for), using B's two-zone page format as the canon throughout. The decision is sequencing.

10The on-disk format — adopt OKF

Google Cloud's Open Knowledge Format (OKF) v0.1, published 2026-06-13 (after the survey above), explicitly "formalizes the LLM-wiki pattern into a portable, interoperable format." It's the file format the redesign should write — independent, week-old validation of nearly every decision in the plan.

What it isA format, not a system — "OKF is a format, not a service or a platform." A vendor-neutral way to write a knowledge base as markdown files + YAML frontmatter so a wiki produced by one tool can be consumed by another agent with no translation. Think "HTML for agent knowledge bases": it standardizes the file shape and says nothing about the brain that grooms them.

Why adopt it (nearly free)

  • Interop: any OKF consumer can read your memory, and you can import OKF bundles (e.g. a data catalog) as memory.
  • A ready-made spec — you stop inventing frontmatter conventions.
  • A free browse UI: OKF ships a reference HTML visualizer that renders a bundle as an interactive graph.
  • External validation of the whole direction, from Google, dated the same week.

What it does NOT give you

  • It's the format, not the brain — same gap as Eve. No write-gating, consolidation, forgetting, ranking, or dreaming. You still build all of §05 yourself.
  • Catalog origin: example bundles are GA4/BigQuery/Stack Overflow; the resource URI assumes each doc describes an asset. For personal memory many atoms have no URI — fine, it's optional.
  • One divergence: OKF keeps history in a single bundle-level log.md; the plan keeps a per-page timeline (better for retrieval). Stay conformant by treating log.md as the bundle changelog (git already gives you this) and the per-page timeline as ordinary concept-doc body.
DecisionAdopt OKF as the on-disk format for the new memory dir. It's the skin, not the organism: write OKF-conformant pages, get interop + a spec + the reference viewer for free, and keep building the maintenance brain on top. Sources: repo + SPEC.md · Google Cloud blog.

11Risks & what the critic flagged

A skeptical pass on the whole package. Two of these are blockers you should resolve before building.

Blocker 1 · security / trust boundary The package has no prompt-injection / memory-poisoning model for an always-on agent that ingests untrusted email/slack/web. "Evolves without manual curation" + an autonomous write path fed by passive sources = a live poisoning vector (see ChatGPT memory attacks). Before building: add a write-time source-trust tag (user | synthesis | integration:<name>) that conflict-resolution respects, a write allow-list of slug prefixes, and a sensitive-content exclude filter (gbrain's exclude_patterns) that runs before any LLM call. Half-present already: source_ref.kind exists.
Recommended (not a blocker) · a tiny eval keeps knobs honest Once you do add tunable knobs (importance thresholds, TTL, a salience floor), they're vibes without a probe set. Cheap fix: hand-write ~20 probe Q/A pairs ("what's my bike setup", "when's the MATS deadline") + a few known-stale/known-superseded cases. Under the one-pass plan there's no "old vs new each night" — instead capture today's answers once before migrating as the baseline, then run the probes against the new store for an absolute pass/fail. It's a 20-minute spot-check, not a gate.

Also unaddressed

  • Obsidian. NTRP's primary source is a markdown+wikilink vault. A file-canonical memory makes a second markdown+[[link]] tree. Decide: (a) memory is a subfolder Obsidian indexes (human gets links into agent knowledge for free), or (b) Obsidian is read-only, never written. Currently ignored.
  • Cost budget. Dream+reflect+retain+health+per-write poignancy vs today's near-zero anti-heartbeat. Set a per-night token/$ ceiling (reuse INTEGRATION_INGEST_MAX_LLM_CALLS) and have HEALTH report actual spend.
  • The topic-page UI (render_html widget) already projects these pages — a file-canonical flip changes what it reads.

Weak claims to keep honest

  • The "field decisively converged on files" leans on days-old 2026 vendor sources (Eve, Copilot/Cursor numbers) that can't be independently corroborated yet. Zep is still shipping graphs.
  • The ACE 18,282→122 collapse figure is one paper's anecdote, repeated as if it were law. Real risk; not a proven universal.
  • "NTRP strictly better than gbrain on generic" undersells gbrain's concept-synthesis tiering (richer emergent structure than NTRP's flat labels).
  • "Brute-force cosine fine to ~10–20k pages" is asserted, not benchmarked. With obsidian+browser+email ingest the count could blow past that.
  • Sequencing tension: Phases 1–2 make the system more complex before simplifying. If you want simple-first, an alternative is to do the A flip earlier — but accept the no-baseline migration risk.

High-value ideas the panel surfaced beyond the core design

  • Provisional status for machine-authored insights: dream-authored insights auto-expire unless retrieved/confirmed within N days (Copilot's 28-day-reset, applied specifically to net-new claims) — contains self-poisoning from the one flow that authors knowledge.
  • RAPTOR/HippoRAG hierarchical summarization as the concrete algorithm for "compress into a tiered map": daily → topic → domain-summary tree, so retrieval enters at the right altitude.
  • Surface "think"-mode gap notes in the UI ("haven't seen X since DATE / two records contradict") — turns blind spots visible and gives a zero-effort correction surface, itself a curation-free evolution mechanism.

12Twelve design principles to hold

  1. Generic and emergent, never CRM. Keep the flat function-typed records + open-vocab labels (you already beat dex here); let page types emerge from clustering your own labels, not a person/company schema.
  2. Pick ONE source of truth. Make markdown canonical; durability via the plain files themselves (append-only timeline + supersede-not-delete + a backup copy before any destructive pass); the vector index is a throwaway derived artifact. This deletes the three-store drift machinery.
  3. Two-zone pages. Compiled prose above, append-only dated timeline below, inline provenance, source precedence (user > compiled > timeline > external).
  4. Never let an LLM rewrite a whole file. Small deterministic deltas merged by code + a GEPA regression guard that rejects collapse and reverts.
  5. Supersede, don't delete; forget by ranking + TTL-by-class + salience floor. Keep tombstone+prune for true dups; add recency/utility decay at retrieval and a class-based TTL with imp≥8 protected.
  6. Build the cross-domain dream. Wire REFLECTION + the idle trigger to the Generative-Agents 3-step on an importance-budget + nightly floor. This is your differentiator.
  7. Score importance at write time; let it drive both ranking and the trigger. Replace the fixed clock with an importance-budget heartbeat (busy days reflect more) + a nightly floor + crash catch-up.
  8. One write chokepoint, one worthiness gate. Remove or converge remember() onto the Curator judge.
  9. Maintenance off the hot path, but reconcile sooner than nightly. Isolated worktree, fresh context; every-N-interactions trigger so append+compress never degrades to append-only.
  10. Resident = small task-relevant synthesis, not a bullet dump. DC-Retrieval-Synthesis; reserve always-inject for a small editable profile; deep recall pull-only.
  11. Self-audit and surface gaps. Add the health/doctor job and a "think" mode that returns cited prose + a gap note. Make blind spots visible.
  12. Delete the accidental complexity. Drop dead tables/columns; shrink artifacts.py; if files become canonical, most regex/slug/changelog plumbing disappears — the "file-based and simple" bar.

13Next steps — just build it

This is a single-user system you own, not a product with users to protect — and you've waited long enough. So the move is not "ship a phase, prove it for weeks, maybe flip later." It's build the end-state directly, in one pass, and delete the old engine. What makes going straight to the target safe isn't caution — it's reversibility.

Guiding stanceBuild the target, not a staircase to it. Files-canonical OKF pages · two-zone format · the cross-domain dream · supersede · source-trust — wired together, migrated in one pass, the old record-store engine retired (the shared transcript index untouched). Forgetting/health/embeddings are deferable — the store is ~80 records, there's nothing to forget yet. No parallel-run-for-weeks, no "earn it," no go/no-go gates that make you wait. The reason you can go straight to the end and not get hurt: memory is just plain files — copy the dir before any destructive pass and the timeline is append-only, so a bad pass is undone by restoring the backup. Be bold; undo is cheap.
The only thing that isn't optionalSpeed is safe because of three cheap properties — and none of them is a gate that delays you, you just build them in: (1) commit per pass → free undo of any mistake; (2) never let the model rewrite a whole file — apply deltas with a sanity check, since a silent whole-file collapse is the one failure that loses data even on a personal system; (3) tag ingested content with its source — and make the dream respect the tag (an insight built only from passive/integration sources stays provisional, never user-trust), so passive email/web can't launder a poisoned line into a confident memory. That's it — everything else is preference, not protection.

One build, roughly in the order you'd type it (not phases, not gates)

  1. Stand up the target tree in a fresh dir. OKF two-zone pages, the always-resident system/ folder, a throwaway index dir (rebuildable, never canonical). Keep the legacy projection and memory.db out of the new page tree (memory.db stays a frozen backup). A timestamped copy of the dir is the safety net that lets everything after it be reckless.
  2. Migrate in one pass — fields mechanically, filing with judgment. Each record's fields map onto a timeline line, but give pinned/scope/freshness a home first (dropping pinned + running forgetting = self-inflicted loss), and label-less/multi-subject records need a filing pass to pick a page. Commit as the baseline. Keep memory.db frozen on disk (it's ~3MB) as a re-migration source so a bad filing pass is cheaply re-runnable, not just revertable-to-before.
  3. Wire the day-one loop. Cheap write-tag on the hot path; off-path consolidate + the cross-domain dream (cited, source-trust-respecting) + supersede. That's the whole spine for ~80 records. Defer forgetting/TTL, a health/doctor job, per-page embeddings, and hierarchical summarization until a real symptom appears — each is an independent later commit, not part of the first pass.
  4. Retire the old engine in the same change — scoped precisely. Drop the source="record" vectors and the records/records_fts tables and the dead projection code; the shared search.db transcript vectors and sessions.db are untouched (deleting them would wipe transcript/context search, unrelated to this migration). Ordering inside this one build: the file-backed index + a file-reading recall() must answer queries before you drop the record vectors, or you get a retrieval blackout. Delete the frozen memory.db in a trivially separate later step once the file store is exercised.

Stances worth fixing so they don't drift mid-build

What evolution means here

Knowledge synthesis (the dream), not behavior/procedural learning. A conscious lazy bound — the agent learns what's true about you and connects it across domains; it does not (yet) learn how to act differently. Revisit only if daily use exposes the gap. Keeps the headline promise honest instead of silently narrowed.

Go straight to files

Don't keep SQLite as a hedge. Maintaining two canonical stores "just in case" is precisely the complexity you're deleting. Files are canonical from the first commit; the DB becomes a disposable index or goes entirely. Git is your hedge — and it's a better one.

How much OKF?

Three frontmatter keys, nothing more. Skip the validator, the import story, and the reference viewer until a real consumer exists. OKF is the skin (free conformance + future interop), not a project. Tooling for an imagined audience on a single-user system is exactly the scope creep to refuse.

Migrate and delete together

The simplification is part of the change, not a reward you defer. Do the migration and the engine deletion in one pass. "Add now, simplify once it's proven" is how the old system never dies — and you've waited long enough to be done with it.

What good feels likeAn unprompted, correct, cited cross-domain insight you didn't ask for; memory you can open in any editor and hand-fix; a corpus that shrinks rather than bloats; and one git'd source of truth that's plainly simpler than the three-store thing you have now. A tiny probe set is a nice 20-minute spot-check that recall didn't regress — use it if you feel like it, skip it if you don't. It's a convenience, never a checkpoint.
