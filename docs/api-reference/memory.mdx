## Mental model

The memory API is exposed under `/admin/memory/*`. The substrate is a flat pool
of atomic **records** with scope metadata for visibility. That flat record pool
is canonical for storage, retrieval, and mutation. User-created records are
typed `directive`, `fact`, or `source`; `changelog` is generated audit output,
not a user-create kind. There is no graph, no lenses, and no derivation edges â€”
record detail returns empty `parents`/`children` for client compatibility only.

Most users should manage memory from the desktop Memory view. These endpoints
back that surface and other admin clients.

## Records

### Create record

Quick-capture a single record into the pool. `kind_tag` is one of
`directive | fact | source`.

```bash
curl -X POST http://localhost:6877/admin/memory/record \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"text":"User prefers concise engineering feedback.","kind_tag":"fact"}'
```

Optional `scope_kind` / `scope_key` set an explicit scope; otherwise the scope
is resolved from the kind (directives are global, others default to user).

### Pin or unpin record

Pinned records survive consolidation and ride along in the resident profile.

```bash
curl -X POST http://localhost:6877/admin/memory/record/rec_123/pin \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"pinned":true}'
```

## Items

Records are returned as `MemoryItem` objects for the desktop client.

### List items

`status` is `active` (default) or `superseded`. Accepts `limit`, `offset`, `q`
(switches to search), and `scope_kind` / `scope_key` / `kind` filters.

```bash
curl "http://localhost:6877/admin/memory/items?limit=50" \
  -H "Authorization: Bearer $API_KEY"
```

### Get item

```bash
curl http://localhost:6877/admin/memory/items/item_123 \
  -H "Authorization: Bearer $API_KEY"
```

### Search items

Hybrid keyword + semantic search over the record pool. Accepts `limit`,
`include_inactive`, and `scope_kind` / `scope_key` / `kind` filters.

```bash
curl "http://localhost:6877/admin/memory/search?q=backend%20preferences&limit=20" \
  -H "Authorization: Bearer $API_KEY"
```

## Scopes

Returns the user-facing scope list. Currently empty â€” scopes are stored and
enforced internally on read/write paths, but are not exposed as a browser.

```bash
curl http://localhost:6877/admin/memory/scopes \
  -H "Authorization: Bearer $API_KEY"
```

## Maintenance

### Prune (LINT)

Trigger the LINT pass on demand: hard-delete superseded records, drop the labels
they orphan, and reconcile the vector index. Runs automatically each
consolidation sweep too.

```bash
curl -X POST http://localhost:6877/admin/memory/prune \
  -H "Authorization: Bearer $API_KEY"
```

## Artifacts

A generated, read-only markdown projection of memory for browsing. Generated
artifacts include `context/`, `references/`, `entities/`, `projects/`,
`facts/index.md`, `directives.md`, `me.md`, `active-work.md`, and `changelog/`.
The `context/` tree is agent-facing context derived from the flat record pool.
`context/integrations/` is generated from existing records and source refs, not
from integration configuration.

### List artifacts

Accepts optional `kind` and `q` query parameters.

```bash
curl http://localhost:6877/admin/memory/artifacts \
  -H "Authorization: Bearer $API_KEY"
```

### Read artifact

```bash
curl http://localhost:6877/admin/memory/artifacts/facts/index.md \
  -H "Authorization: Bearer $API_KEY"
```

### Rebuild artifacts

Regenerate the markdown projection from the current records.

```bash
curl -X POST http://localhost:6877/admin/memory/artifacts/rebuild \
  -H "Authorization: Bearer $API_KEY"
```
